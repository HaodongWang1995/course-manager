# 新增需求

与用户对话，收集需求信息，创建 PRD 文档。

## 流程

### 步骤 1 — 了解需求

与用户对话，逐步了解：
- 功能名称
- 功能描述和使用场景
- 目标用户（教师/学生/两者）
- 验收标准（完成的定义）
- 优先级

保持简洁，不要一次问太多。

### 步骤 2 — 确认需求

整理后向用户确认：
```
【需求确认】
- 功能名称：xxx
- 优先级：P0/P1/P2
- 描述：xxx
- 验收标准：
  1. xxx
  2. xxx

是否准确？有需要补充的吗？
```

用户确认后才继续。

### 步骤 3 — 分析技术方案

阅读当前项目代码，分析：
- 涉及哪些文件/模块需要修改
- 是否需要新增 API 端点或数据库迁移
- 是否需要新增 UI 组件
- 技术选型建议（如有多种方案，列出对比）

将技术方案写入 PRD 文件的「实现方案」章节。

### 步骤 4 — 创建 PRD 文件

在 `PRD/` 目录下创建文件，命名规则：`未开始-{需求名}-{YYYYMMDD}.md`

文件模板：

```markdown
# 需求：{需求名}

## 基本信息
- **状态**：未开始
- **优先级**：{P0 | P1 | P2}
- **创建日期**：{YYYY-MM-DD}
- **最后更新**：{YYYY-MM-DD}

## 需求描述

{详细描述}

## 验收标准

- [ ] {条件1}
- [ ] {条件2}

## 实现方案

### 涉及文件
- {文件路径}

### 数据库变更（如需）
- {迁移描述}

### API 变更（如需）
| 方法 | 路径 | 说明 |
|------|------|------|

### 前端变更
- {组件/页面描述}

## 任务清单

- [ ] {子任务1}
- [ ] {子任务2}

## 备注

{其他说明}
```

### 步骤 5 — 提交 git

```bash
git add PRD/
git commit -m "docs(PRD): 新增{需求名}需求文档"
```

完成后告知用户文件路径。
